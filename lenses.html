Architectural Lenses

Lenses represent different architectural concerns, providing distinct perspectives through which to view a system. Each lens allows stakeholders to focus on specific aspects without being overwhelmed by unrelated details. Below is the set of lenses proposed by this framework.

  • System Lens

    What the system is and how it is structured. Covers boundaries, context, containers, components, and their relationships.

  • Capability Lens

    What the system enables the business to do. Describes business capabilities, value streams, feature groupings, and ownership domains.

  • Domain Lens

    The conceptual model of the problem space. Core domain concepts, aggregates, bounded contexts, ubiquitous language, invariants.

  • Data Lens

    How information is structured, stored, and flows. Data models, ownership, lifecycle, integration flows, persistence choices, APIs, schemas.

  • Integration Lens

    How the system interacts with other systems. APIs, contracts, events, external dependencies, messaging, protocols, versioning strategy.

  • Runtime Lens

    How the system behaves when it runs. Execution flows, concurrency, scaling, state management, failure handling, resilience patterns.

  • Deployment Lens

    How the system is built, deployed, and operated. Environments, infrastructure topology, CI/CD, containerization, cloud setup, networking.

  • Security Lens

    How risk is mitigated and trust is established. Threat model, identity, authorization, data protection, compliance, auditability.

  • Quality Attributes Lens

    How non-functional requirements are realized. Performance, availability, scalability, maintainability, observability, operability.

  • Governance & Decision Lens

    Why the architecture is the way it is. Architecture Decision Records (ADRs), constraints, principles, standards, trade-offs.

  • Evolution Lens

    How the system changes over time. Roadmap, technical debt, migration paths, deprecation strategy, architectural runway.

  • UX & Interaction Lens

    How the user perceives and interacts with the system. Covers user journeys, design systems, frontend architecture patterns, and accessibility.

  • Organization Lens

    How the human organization aligns with the technical architecture. Covers team topologies, ownership boundaries, and cognitive load management.

  • Cost Lens

    The economic perspective of the architecture. Covers infrastructure spend, licensing, operational costs, and environmental sustainability.

  • Compliance Lens

    The regulatory and legal perspective. Covers industry standards (SOC2, HIPAA), data residency laws, and audit requirements.
