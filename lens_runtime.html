Runtime Lens

Description: How the system behaves when it runs. Execution flows, concurrency, scaling, state management, failure handling, resilience patterns.

General Level Guidance

Level 1 — Introduction

  • How does the system behave when it is 'alive'?
  • Is it primarily synchronous, asynchronous, or batch-oriented?
  • What is the high-level request journey through the system?

Common Fulfillment: A high-level execution flow diagram or 'Request Journey' narrative.

Level 2 — Orientation

  • How is traffic routed through the system components?
  • How does the system scale (or fail) under load?
  • What is the concurrency and threading model?

Common Fulfillment: Links to core Monitoring Dashboards (e.g., Grafana, Datadog) and scaling policy definitions (HPA).

Level 3 — Elaboration

  • How exactly does the system handle failures (e.g., Circuit Breakers, Bulkheads)?
  • What are the specific resource limits (CPU/Memory) and GC settings for the processes?
  • How is distributed tracing or request-id propagation implemented?

Common Fulfillment: Link to Distributed Tracing tools (e.g., Jaeger, AWS X-Ray) and specific runtime configurations (e.g., application.yaml, pod-spec.yaml).

Example: Completely over-engineered Todo-App

Level 1 — Introduction

The Todo system is a highly concurrent, stateless microservices environment designed to scale horizontally across multiple availability zones under varying user loads.

Level 2 — Orientation

The runtime behavior is managed by Kubernetes. Requests follow a path from the user's browser to CloudFront (CDN), then through an Application Load Balancer (ALB) to the EKS cluster. We use Ingress-NGINX as our in-cluster controller to route traffic to specific service pods.

Scaling is reactive: the Horizontal Pod Autoscaler (HPA) monitors CPU and Memory metrics, adding replicas when 5-minute averages exceed 70%. All services are designed to be stateless, storing session data in Redis to allow any pod to handle any request.

Level 3 — Elaboration

Request Execution Flow (Hot Path)

A typical GET /tasks request involves:

  1. ALB Terminates TLS 1.3.
  2. Kong Gateway performs JWT validation.
  3. Kong forwards to Todo Service Pod via internal K8s Service DNS.
  4. Todo Service checks Redis for a cached response (TTL 60s).
  5. If cache miss, Todo Service queries RDS PostgreSQL.
Resilience & Failure Modes

We implement the Circuit Breaker pattern (via Resilience4j or similar) between the Todo Service and the Database. If RDS latency spikes above 2s for 5 consecutive requests, the circuit opens and the service returns a 503 Service Unavailable or a cached partial response, preventing pod resource exhaustion.

Concurrency & Thread Model

The Real-time Sync Engine (Elixir/Phoenix) uses the Erlang VM's 'Actor Model' to manage 50,000+ simultaneous WebSocket connections per node. The Todo Service (Node.js) uses a single-threaded non-blocking event loop, which is why we strictly offload CPU-intensive tasks (like JSON schema validation for large lists) to a background worker pool.

Resource Management per Pod
  • Todo Service: 256Mi Memory Request, 512Mi Limit. GC tuning optimized for 'throughput' over 'latency'.
  • User Service: 128Mi Memory Request, 256Mi Limit.
