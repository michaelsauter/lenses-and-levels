Deployment Lens

Description: How the system is built, deployed, and operated. Environments, infrastructure topology, CI/CD, containerization, cloud setup, networking.

General Level Guidance

Level 1 — Introduction

  • Where does the system live physically or virtually (Cloud, On-prem, Hybrid)?
  • How many distinct environments exist (e.g., Dev, Staging, Prod)?
  • What is the high-level deployment philosophy (e.g., Immutable, Blue/Green, Canary)?

Common Fulfillment: A simple list of environments and a cloud provider high-level account map.

Level 2 — Orientation

  • What does the network topology look like (Subnets, VPCs, Firewalls)?
  • What are the stages of the CI/CD pipeline?
  • How is infrastructure defined and synchronized (e.g., Terraform, GitOps)?

Common Fulfillment: A Network Topology diagram and links to CI/CD pipeline definitions (e.g., GitHub Actions, GitLab CI).

Level 3 — Elaboration

  • What are the specific Infrastructure-as-Code definitions or Helm charts?
  • How are secrets and environment-specific configurations injected at runtime?
  • What is the specific automated roll-back and disaster recovery procedure?

Common Fulfillment: Direct link to Terraform/Pulumi/Bicep source code, Helm charts, or Kubernetes manifests.

Example: Completely over-engineered Todo-App

Level 1 — Introduction

The Todo platform is a multi-region AWS deployment managed via GitOps, primarily running on Amazon EKS with RDS for persistence.

Level 2 — Orientation

Our infrastructure follows a strict GitOps model. Changes to the EKS cluster and AWS resources are defined in Terraform and deployed via a centralized GitHub Actions runner. Application-level Kubernetes resources (Deployments, Services, ConfigMaps) are managed by ArgoCD, which synchronizes state between our Git 'Manifest Repo' and the cluster.

The eu-central-1 region is our primary site, with resources spread across three Availability Zones (AZs) for high availability. We maintain three distinct environments: Development (unstable, latest branch), Staging (pre-production mirror), and Production.

Level 3 — Elaboration

CI/CD Pipeline Architecture
  1. Commit: Developer pushes to main in a service repo.
  2. Build: GitHub Action runs unit tests and builds a Docker image via Kaniko.
  3. Push: Image is pushed to Amazon ECR with a unique SHA tag.
  4. Update: GitHub Action updates the image tag in the k8s-manifests repo.
  5. Sync: ArgoCD detects the manifest change and performs a 'Rolling Update' in the EKS cluster.
Network Topology & Security Groups

We use a custom VPC with 10.0.0.0/16 CIDR. Services sit in Private Subnets and can only be reached via the Public ALB. RDS is isolated in a dedicated DB-Subnet-Group with ingress restricted strictly to the EKS Node Security Group on port 5432.

Kubernetes Resource Quotas

To prevent a single runaway service from crashing the cluster, we enforce Namespace-level ResourceQuotas. The todo-production namespace is limited to 16 vCPUs and 32Gi of RAM total across all pods.

Secrets Management

We use the External Secrets Operator. Secrets are stored in AWS Secrets Manager and automatically mapped into Kubernetes Secrets as opaque data, which are then mounted as environment variables in the service pods. This ensures no sensitive data ever touches the Git repository.
