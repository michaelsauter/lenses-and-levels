Integration Lens

Description: How the system interacts with other systems. APIs, contracts, events, external dependencies, messaging, protocols, versioning strategy.

General Level Guidance

Level 1 — Introduction

  • Which external systems does this system depend on?
  • Which external systems depend on us?
  • What is the primary business purpose of these integrations?

Common Fulfillment: An 'External Dependencies' table or a simple integration overview diagram.

Level 2 — Orientation

  • What protocols and standards are used (e.g., REST, GraphQL, Async API)?
  • Where can developers find the API or Event catalog?
  • What is the global versioning and deprecation strategy?

Common Fulfillment: Link to an API Portal (e.g., Backstage, SwaggerHub) or an Event Catalog (e.g., EventBridge Registry).

Level 3 — Elaboration

  • What are the specific endpoint contracts, event schemas, or WSDLs?
  • How are retries, timeouts, and partial failures handled for this specific integration?
  • What are the specific authentication, authorization, and rate-limiting rules?

Common Fulfillment: Direct link to Swagger/OpenAPI/AsyncAPI definitions or Postman collections.

Example: Completely over-engineered Todo-App

Level 1 — Introduction

The Todo platform integrates with Auth0 for identity management and SendGrid for transactional email, while exposing a public v1 REST API for ecosystem partners.

Level 2 — Orientation

We leverage a multi-protocol integration strategy. Public traffic is strictly REST/JSON via the API Gateway. Internal service-to-service integration is Event-Driven using AWS SNS and SQS for decoupling, or gRPC for high-performance synchronous calls (e.g., Auth validation).

Our API Catalog is hosted on a SwaggerHub instance, and we use EventBridge as our central event bus for domain-wide broadcasts like Task.Completed or Subscription.Renewed.

Level 3 — Elaboration

Public API Endpoint Specification

POST /v1/lists/{id}/tasks creates a new task. Request payloads must be valid JSON matching the NewTaskSchema. We implement Rate Limiting at the Kong Gateway level: 500 requests per minute per IP for authenticated users.

Event Schema & Dead Letter Strategy

Integration events follow the CloudEvents spec. For example, com.todoapp.productivity.task.assigned contains the assignee_id and list_id. If the Notification Service fails to process an event, it is moved to an SQS Dead Letter Queue (DLQ). After 3 retries with exponential backoff, a manual intervention alert is raised via OpsGenie.

Auth0 OIDC Integration Flow
  1. Frontend redirects to auth.todoapp.com.
  2. Successful login returns an Authorization Code.
  3. Backend exchanges code for an Access Token and Refresh Token.
  4. The User Service synchronizes the Auth0 sub claim with our internal user_id.
API Versioning & Lifecycle

We use URL-based versioning (/v1/). A version is marked 'Deprecated' when a newer version is released. Deprecated versions are maintained for 12 months with security patches only before being 'Sun-setted' (terminated).
