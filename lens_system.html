System Lens

Description: What the system is and how it is structured. This lens pairs extremely well with the C4 Model for visual decomposition.

General Level Guidance

Level 1 — Introduction

Corresponds to the C4 System Context Diagram. Focus on the high-level structural position of the system in the world.

  • Who are the primary human actors (People) that use the system?
  • Which other Software Systems does this system interact with?
  • What are the high-level responsibilities of the system?

Common Fulfillment: A high-level C4 System Context diagram and a concise description of the system's role.

Level 2 — Orientation

Corresponds to the C4 Container Diagram. Focus on the high-level technical decomposition into deployable units.

  • How is the system divided into autonomous, deployable units (Containers)?
  • What are the primary responsibilities of each container?
  • Which technology stack (language, framework, data store) is used for each unit?
  • How do these containers communicate with each other and external systems?

Common Fulfillment: A C4 Container diagram showing the shape of the architecture.

Level 3 — Elaboration

Corresponds to the C4 Component Diagram. Focus on the internal building blocks of a specific container.

  • What are the internal building blocks (Components) of a specific container?
  • What is the responsibility of each component and its implementation technology?
  • What are the internal and external relationships between these components?

Common Fulfillment: C4 Component diagrams for the most critical or complex containers.

Example: Completely over-engineered Todo-App

Level 1 — Introduction (System Context)

The Todo Application is a cloud-native platform sat within the AWS ecosystem that allows users to manage tasks across web and mobile devices via a centralized API.

Level 2 — Orientation (Containers)

The system is architected as a set of autonomous microservices deployed into an Amazon EKS cluster. The frontend is a React-based Single Page Application (SPA) served via CloudFront, which communicates with the backend through a shared Kong API Gateway.

Key containers include the Todo Service (Node.js) for core task logic, the User Service (Go) for profile management, and a Real-time Sync Engine (Elixir) for WebSocket-based task updates. Each service is contained in its own Git repository and has its own isolated PostgreSQL database schema.

Level 3 — Elaboration (Components)

Todo Service Internal Decomposition

The Todo Service is structured using Hexagonal Architecture (Ports and Adapters) to ensure business logic remains independent of infrastructure. The core domain handles task state transitions (e.g., 'In Progress' to 'Completed') and recurring task generation logic.

Interaction Flow: Task Completion

When a user marks a task as done, the following sequence occurs:

  1. API Gateway validates the JWT and forwards the PATCH request to the Todo Service.
  2. The TaskController invokes the CompleteTask Command Handler.
  3. The Command Handler fetches the TaskAggregate, performs validation, and marks it done.
  4. The PostgreSQLAdapter persists the change.
  5. An IntegrationEvent is published to the TaskCompleted SNS topic for the Sync Engine.
Code Organization Conventions

All services must follow the /cmd, /internal/domain, and /internal/infrastructure directory pattern. Inter-service communication is strictly via EventBridge or gRPC; direct synchronous REST calls between backend services are forbidden to prevent cascading failures.
