Security Lens

Description: How risk is mitigated and trust is established. Threat model, identity, authorization, data protection, compliance, auditability.

General Level Guidance

Level 1 — Introduction

  • What is our primary security philosophy (e.g., Zero Trust, Defense in Depth)?
  • What are the most critical assets we are protecting?
  • Who are our primary threat actors and what are we worried about?

Common Fulfillment: A high-level Security Strategy statement or 'Top 5 Security Goals' list.

Level 2 — Orientation

  • How is identity and access managed globally (AuthN/AuthZ)?
  • Where are the trust boundaries located in the system?
  • How is data classified and where is encryption applied (Transit vs Rest)?

Common Fulfillment: A high-level Identity Architecture diagram and links to Secrets Managers (e.g., HashiCorp Vault, AWS Secrets Manager).

Level 3 — Elaboration

  • What is the specific RBAC, ABAC, or Policy-as-Code matrix?
  • What are the detailed threat models for the highest-risk components?
  • How are audit logs and security events collected and protected for compliance?

Common Fulfillment: Detailed Threat Model documents (e.g., STRIDE analysis), OPA/Rego policy files, or RBAC YAML definitions.

Example: Completely over-engineered Todo-App

Level 1 — Introduction

Our security posture is Zero Trust by design, assuming the network is always compromised and requiring explicit cryptographic proof for every interaction.

Level 2 — Orientation

The identity perimeter is managed by Auth0 (OIDC) for human users and AWS IAM (IRSA) for service-to-service authorization. Within the EKS cluster, Kubernetes Network Policies create a default-deny posture, strictly whitelisting only the necessary traffic paths (e.g., Gateway -> Todo Service).

Data is classified into three tiers: Public (Static assets), Private (User task titles), and Sensitive (Email addresses, Session tokens). All 'Private' and 'Sensitive' data is encrypted at rest using AWS KMS-managed keys and in transit via mTLS within the cluster mesh.

Level 3 — Elaboration

Detailed Threat Model: Task Disclosure

We've identified 'Unauthorized Task Access' as a primary threat. Mitigation involves:

  • Identity Validation: JWT must contain a valid sub claim.
  • Application Authorization: Todo Service performs a row-level check: SELECT * FROM tasks WHERE id = ? AND list_id IN (SELECT list_id FROM user_permissions WHERE user_id = ?).
  • Infrastructure Protection: Database is in a private subnet with no public IP.
Encryption Implementation Specs

We use AES-256-GCM for application-level encryption of 'Sensitive' fields before they hit the database. The encryption keys are rotated every 90 days via KMS automated rotation. TLS 1.3 is enforced at the ALB with Perfect Forward Secrecy enabled.

Audit Logging Architecture

All administrative actions (e.g., User Deletion, Role Grant) are logged via a sidecar to CloudWatch Logs. These logs are then exported to a restricted S3 bucket with Object Lock enabled to ensure immutability for compliance audits.
