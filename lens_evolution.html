Evolution Lens

Description: How the system changes over time. Roadmap, technical debt, migration paths, deprecation strategy, architectural runway.

General Level Guidance

Level 1 — Introduction

  • What is the long-term vision for the architectural state (3-5 years)?
  • Which major technological or structural shifts are currently in progress?
  • What is the overall philosophy for managing technical debt and obsolescence?

Common Fulfillment: A high-level vision statement and a 'Current vs. Target State' roadmap overview.

Level 2 — Orientation

  • What is the concrete architectural roadmap for the next 12 months?
  • Which components are currently 'Stable', 'In-Migration', or 'Deprecated'?
  • Where is the central registry for tracking and grooming technical debt?

Common Fulfillment: A visual roadmap and a direct link to the Technical Debt Register (e.g. Jira/Linear board).

Level 3 — Elaboration

  • What is the step-by-step procedure for safely migrating critical data or traffic?
  • How exactly do we sunset and decommission legacy components (e.g. 'Kill-switch' protocols)?
  • What specific 'Architectural Runway' projects are active to enable future features?

Common Fulfillment: Detailed migration plans, decommissioning checklists, and RFCs for future-enabling infrastructure.

Example: Completely over-engineered Todo-App

Level 1 — Introduction

The Todo platform is transitioning from a Monolithic Java backend to a Go/Node.js Microservices architecture to support our 10x growth target.

Level 2 — Orientation

Our evolution strategy centers on the Strangler Fig Pattern. We are gradually extracting business capabilities into new services while keeping the legacy monolith as a 'Downstream' system for un-migrated data. Our 12-month roadmap targets the decommissioning of the legacy Oracle database by Q4 2026.

A major focus is the Technical Debt Register, which currently prioritizes the 'Auth Token Refactor' and 'Legacy CSV Export' migration. These items are tracked in Jira and groomed monthly by the Architecture Guild.

Level 3 — Elaboration

Data Migration Plan: Monolith to PostgreSQL

The migration of 10M+ user lists involves a zero-downtime four-phase approach:

  1. Dual-Write: New microservices write to both the old database and new PostgreSQL schema. Old DB remains the source of truth for reads.
  2. Backfill: A background batch job migrates all historical records that were created before dual-write started.
  3. Shadow Read: Services read from PostgreSQL but validate against Oracle in the background, logging any discrepancies.
  4. Cut-over: PostgreSQL becomes the source of truth. Oracle writes are disabled.
Service Sunsetting Procedure

When a legacy component is ready for decommissioning, we follow the DECOMM-01 protocol:

  • Remove all ingress traffic from the ALB.
  • Archive the Git repository and mark as Read-Only.
  • Wait 30 days for any 'silent' batch dependencies to surface.
  • Purge all related container images and EBS volumes.
Architectural Runway: AI-Assisted Tasks

To support future 'Smart Schedule' features, we are currently deploying a Vector Database (Pinecone) as part of our architectural runway. This ensures that when the ML team is ready to implement task embeddings, the necessary storage and retrieval infrastructure is already hardened and integrated into our deployment pipelines.
