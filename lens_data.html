Data Lens

Description: How information is structured, stored, and flows. Data models, ownership, lifecycle, integration flows, persistence choices, APIs, schemas.

General Level Guidance

Level 1 — Introduction

  • What are the major categories of data the system handles (e.g., Transactional, Analytical, Logs)?
  • Where does the "Source of Truth" live for critical entities?
  • What is the high-level privacy posture and data sensitivity (e.g., "We handle PII and must comply with GDPR")?

Common Fulfillment: A high-level Information Map or conceptual Data Flow Diagram.

Level 2 — Orientation

  • Which persistence technologies are used for which data types?
  • How is data classified by sensitivity (e.g. Public, Internal, Restricted, PII)?
  • Which specific system components (from the System Lens) handle or own sensitive data?
  • How does data move between storage systems (e.g., CDC, ETL, Events)?

Common Fulfillment: A Data Landscape diagram and a Data Classification matrix.

Level 3 — Elaboration

  • What are the specific table schemas, document structures, or event formats?
  • What is the indexing, partitioning, or sharding strategy?
  • What are the specific technical implementations for privacy (e.g. Anonymization, Encryption at field-level, Deletion protocols)?

Common Fulfillment: Logical ERD, SQL DDL scripts, or JSON/Avro schema definitions. Link to privacy impact assessments (PIA).

Example: Completely over-engineered Todo-App

Level 1 — Introduction

The application manages Relational Task Data for core functionality and Time-series Audit Streams for user history. We adopt a Privacy-First posture, minimizing the collection of PII to only what is required for authentication.

Level 2 — Orientation

Our data stack is split by access pattern and sensitivity. Amazon RDS (PostgreSQL) serves as the system of record. Data is classified as PII/Restricted (User email, Profile), Private (Task content), and Public (Static assets). The User Service is the only component with direct access to PII datasets.

We implement a 'Hot/Cold' data lifecycle: all tasks completed more than 180 days ago are automatically moved from RDS to an S3-based data lake in Parquet format to maintain query performance on the main database.

Level 3 — Elaboration

Entity Relationship Model (ERD)

The core schema revolves around the task_lists table which has a 1:N relationship with tasks. A collaborators join table links users to task_lists with associated permission enums.

Data Retention & Privacy Implementation

Per GDPR requirements, the User Service implements a 'Right to be Forgotten' flow. Deleting a user triggers an async job that:

  1. Scrubs the users table of PII.
  2. Nullifies owner_id in related lists.
  3. Anonymizes task contributions to maintain team continuity without identifying the original author.
Encryption at Rest

Restricted fields (like the email column in the users table) are encrypted at the application layer using AWS KMS before being stored in PostgreSQL, ensuring that even a database dump would not leak PII without access to the specific KMS key.
